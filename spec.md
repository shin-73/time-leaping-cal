# Time-Leap Cal 仕様書

「Time-Leap Cal」は、過去から未来への「時代の空気感」をポストカードのように切り出し、個人の歴史と交差させるクロノロジカル・トランジション・システムです。

## 1. コア・コンセプト
- **AIエディター・ナラティブ**: 単なるデータの羅列ではなく、その時代の情景を映画の一場面のように描写する物語生成。
- **究極のミニマリズム**: 装飾を極限まで削ぎ落とし、言葉と写真、そして贅沢な余白で構成されるマガジン・スタイル。

## 2. 機能仕様
### 2.1 時代検索（タイムリープ）
- **対応元号**: 西暦 (AD)、明治、大正、昭和、平成、令和。
- **動的プレースホルダー**: 選択された元号が「もし現代（2026年）まで続いていたら何年か」を自動計算して表示（例：昭和なら101）。
- **入力最適化**: モバイル端末では自動的にテンキー（数字キーボード）を起動。
- **入力ガード**: 全角数字は半角へ正規化し、数字以外の文字は即時除去する。入力は最大4桁に制限する。
- **SP遷移補完**: iOSテンキーの「完了（Done）」押下時は `onBlur` を起点として検索処理を発火させる。
- **範囲バリデーション**: 1868年未満はエラー表示で遷移を拒否する。

### 2.2 コンテンツ生成（AIナレッジベース）
Wikipediaの生データ解析を廃止し、内部の歴史ナレッジベースによって、見出しを伴わない単一の没入型ナラティブを生成する。
- **生成方針**: 歴史的事実の羅列ではなく、五感（視覚・聴覚・触覚・嗅覚）を含む情緒的な情景描写を優先する。
- **出力形式**: 「時代の情景」等のラベルを出さず、連続した本文として出力する。

### 2.3 パーソナライズ（あなたの歴史）
- **誕生日設定**: ユーザーの誕生日を保存（Local Storage）。
- **SP誕生日入力UI**: モバイル設定画面の誕生日入力は `YYYYMMDD` 8桁のテンキー入力を採用し、保存時は `YYYY-MM-DD` 形式へ正規化する。
- **SP誕生日入力レイアウト**: モバイル設定画面の誕生日入力欄は保存ボタンと同一幅で表示する。
- **ライフステージ表示**: 検索した年にユーザーが何歳で、どのような人生の段階（乳幼児、小学生、社会人など）にいたかを算出。
- **表示順序**: 年齢に関する文は、本文ナラティブの表示完了後に同一演出で末尾へ続けて表示する。
- **表示条件**: ナラティブ生成エラー時（「記憶にアクセスできませんでした。」表示時）は年齢文を表示しない。
- **速度制御**: 年齢文パートのみ本文より遅く表示する（通常文字は待機時間 `x3.0625`、読点「、/ ,」は待機時間 `x2.25`）。

## 3. デザイン・UI/UX
### 3.1 レイアウト
- **1カラム構造**: `max-w-2xl`（約700px）の中央寄せ1カラムに統一。
- **シームレスな遷移**: トップページの入力欄と結果画面の年号のサイズ（Q数）・位置を完全に同期。
- **レスポンス**: iPhone 16e等のコンパクトなデバイスでも要素が改行されない最適化。
- **元号セレクタ配置**: 西暦〜令和の6ボタンは PC/SP ともに常に1行表示を維持する（折り返し禁止）。
- **間隔設計**: 元号ボタン同士は誤タップ防止のため明確な間隔を確保し、近接しすぎないこと。
- **SP設定/ヘルプ導線**: モバイルでは設定（歯車）とヘルプ（？）はモーダルではなくフルスクリーンのページ遷移風表示を採用する。
- **戻る挙動整合**: モバイルの設定/ヘルプ画面はURLクエリと履歴に連動し、ブラウザ戻るで開閉状態を正しく復元する。
- **戻るボタン配置統一**: モバイル結果画面の戻るボタン位置は設定/ヘルプ画面の戻るボタン位置と揃える。
- **戻るボタン背景**: PC/SPともに戻るボタンの背景矩形を使用しない。

### 3.2 タイポグラフィ
- **年号表示サイズ**: 検索欄および結果画面の西暦は `text-5xl sm:text-7xl md:text-[9rem]` を基準とし、表示の迫力を維持しつつ可読性を優先する。
- **本文フォント基準**: 日本語本文・UIラベル・説明文は `Noto Sans JP` を標準とし、全体に `font-stretch: 90%` の長体を適用する。
- **タイトル/数字フォント運用**: サイトタイトルは `Shippori Mincho`、数字（西暦入力・結果年号・日付入力）は `Playfair Display`（weight `400` 固定）を適用する。
- **フォント配信整合**: 実装で使用する `font-weight` は配信済みGoogle Fontsのweight範囲内に限定し、未配信weight（例: `900`）への依存を禁止する。
- **サブタイトル調整**: `Chronological Transition System` は `Noto Sans` で表示し、文字間をわずかに広げ、色は `black/80` を基準とする。
- **ロゴ字間調整**: `Time Leap Cal` は `E` と `L` の間に `1px` 相当の字間を与えて視認性を整える。
- **可読性**: 見出しを排した本文中心の誌面で、余白と行間による可読性を維持する。
- **元号セレクタ文字**: SP版の元号ボタンラベルは視認性を優先し、`text-sm` を基準とする。
- **ラベル折返し防止**: 元号ボタンラベルは `whitespace-nowrap` を維持し、ボタン内改行を禁止する。
- **タップ領域**: 元号ボタンは最低44px高のタップ領域を確保する。
- **SP誕生日入力フォント**: モバイル設定画面の誕生日入力数字は情緒演出を適用せず `Noto Sans` を使用する。
- **SP誕生日入力整列**: モバイル設定画面の誕生日入力値はフォーム中央に表示する。
- **背景適応オーバーレイ**: Hero背景画像の平均輝度を読み取り、年号可読性を維持するため白オーバーレイ濃度を動的調整する。
- **スクロール制御**: ページ遷移（表示切り替え）時に自動的に最上部へスクロール。

### 3.3 UI/UX演出（ナラティブ表示）
- **タイピング表示**: 生成ナラティブは全文を一括表示せず、一文字ずつ段階的に表示する。
- **自然なリズム制御**: 句読点（。、！？）や読点（、）で短い待機を入れ、機械的な等速表示を避ける。
- **反芻の余白**: 文末（。！？）や段落切り替え（改行）では、通常より長い待機を入れて読後の間を作る。
- **段落レイアウト**: 句点（。！？）ごとに改行し、段落間に適度な余白を設ける。
- **統一表示**: 本文ナラティブと年齢文は同じエリア・同じ文字スタイルで連続表示する（装飾罫線なし）。
- **本文フォント演出**: ナラティブ本文（ローディング文含む）は `Shippori Mincho` を適用し、本文領域のみ `font-stretch: normal` として長体を解除する。
- **本文タイピング速度**: 本文ナラティブ（誕生年連動文を除く）は、基準タイピング速度より遅く表示する（待機時間 `x1.5` / 速度 `2/3`）。
- **SP余白最適化**: ナラティブ領域の上余白はモバイル時に従来の約 `2/3` へ圧縮し、ヒーロー画像との距離を短縮する（`pt-10` 基準）。
- **PC余白最適化**: ナラティブ領域の上余白はPC時に従来値から約 `1/2` へ圧縮し、ヒーロー画像との距離を短縮する（`md:pt-16` 基準）。
- **文字間の安定表示**: ナラティブ本文・ローディング文ともに `tracking` や `text-justify` による均等割付を適用せず、表示途中で文字間が広がって見える挙動を防止する。
- **ローディング演出**: ローディング文言は固定表示ではなく、一文字ずつのタイピングで順次表示する。
- **ローディング速度・視認性**: ローディング文言は本文タイピングより大幅に遅く表示する（基準待機時間 `x4`）、かつ本文より薄い文字色（`text-black/55`）で読み分け可能にする。
- **アクセシビリティ**: `prefers-reduced-motion` が有効な環境ではタイピング演出を抑制し、即時表示に切り替える。

### 3.4 エラーメッセージ仕様
- **生成失敗時文言**: `${year}年（${eraName}）の記憶にアクセスできませんでした。` を表示する。

## 4. 技術スタック
- **Frontend**: React, TypeScript, Vite.
- **Styling**: Tailwind CSS (Vanilla CSS approach).
- **スタイル記述方針**: スタイル定義は `src/index.css` などのCSSへ集約し、JSX内のインライン `style` 指定は原則使用しない。
- **Icons**: Lucide React.
- **API**: Google Gemini（`gemini-flash-latest`）+ Wikimedia Commons API（背景画像の動的取得）。
- **Persistence**: URLパラメータによる状態保持、ブラウザの「戻る」ボタンへの完全対応。
- **キャッシュ**: ナラティブはセッション中メモリキャッシュのみを使用し、永続化は行わない。

## 5. 現在の制約・特記事項
- **データ範囲**: 1868年（明治元年）から現在、および数年先までの未来に対応。
- **画像取得**: 検索した西暦に基づき、歴史的な文脈に沿った画像を自動選別して表示。
- **APIタイムアウト**: ナラティブ生成リクエストは25秒で打ち切り、失敗時の無限待機を防止する。
- **接続方式**: 本番環境は Vercel Serverless Function（`/api/generate-narrative`）経由でGeminiへ接続し、APIキーをサーバー側で保持する。開発環境は `import.meta.env.VITE_GEMINI_API_KEY` による直接接続を許容する。
- **モデル保守方針**: `gemini-1.5-flash` を第一候補、`gemini-flash-latest` をフォールバック候補として順次試行する。
- **リトライ方針**: 標準プロファイルでは再試行を行わず、各モデル1回試行で次候補モデルへフォールバックする。

## 6. 通信アーキテクチャの動作保証条件
### 6.1 Serverless経由を必須とする理由（本番）
- **CORS/実行環境差異の回避**: ブラウザ直通信は本番オリジン条件で pending・Timeout 化するリスクがあるため、Vercel Serverless Functionを単一の接続点として安定化する。
- **APIキー秘匿**: Gemini APIキーをクライアントバンドルへ露出させず、サーバー側環境変数でのみ保持する。
- **障害切り分け**: ブラウザ↔自前API と 自前API↔Gemini を分離し、失敗箇所を即時判定可能にする。

### 6.2 環境変数管理
- **クライアント参照ルール**: `VITE_` プレフィックス付き変数のみ `import.meta.env` で参照可能。`process.env` はブラウザ実行コードで使用しない。
- **本番必須項目**: Vercel Production環境に `VITE_GEMINI_API_KEY` を設定し、変更時は必ずRedeployする。
- **サーバー側参照**: `api/generate-narrative` では `process.env.VITE_GEMINI_API_KEY` を参照し、未設定時は明示的に500エラーを返す。
- **動作保証条件**: 本番Networkタブで Gemini直通信ではなく `/api/generate-narrative` のみが呼ばれる状態を正とする。

## 7. デプロイ運用ルール
- **本番反映フロー**: 本番反映は GitHub `main` への push を起点とした Vercel 自動デプロイを標準運用とする。
- **手動デプロイ方針**: `vercel --prod` の手動実行は障害時の例外手段とし、通常運用で常用しない。
- **反映完了定義**: Vercel上のデプロイ成功に加え、本番URLで主要導線（年検索・ナラティブ表示・API疎通）を確認して完了とする。
